## Exact Concept Definitions Used In This Workflow

Before this workflow can produce a correct brief, the agent must reason using
these precise definitions — not generic SEO interpretations.

**Topical Authority**
Ranking higher for a certain amount of time compared to other web sources with higher
clarity, accuracy, information quality, and responsiveness for a cheaper search engine
evaluation and retrieval process, so that the web source is preferred by the search
engine for certain topics for prioritized rankings.
Formula: Topical Coverage × Historical Data.
NOT "focus on topics instead of keywords."

**Historical Data**
Not time. Amount of user engagement for a certain amount of time. Fresh historical
data is more important than older historical data. Includes hover-overs, impressions,
clicks, touch, text-select, clickless searches, and query session logs. Non-quality
engagement can demote a website. Current state reflects historical data from at least
6 months ago.

**Topical Coverage**
Cannot be increased by stuffing entities, attributes, or opening new pages for every
topic. Requires creating a topical border first to calculate coverage. Involves
connected attributes, knowledge domains, and contextual domains for a specific
query network. Requires processing every macro and micro context for a knowledge
domain by connecting and associating every entity via their attributes — including
all content formats, with proper web page layout and components for signalling
related and possible search activities from query context, aspect, and definition
based on query semantics.
If you didn't DEFINE X — you didn't cover it.
If you didn't CONNECT X to Y — you didn't cover it.
If you didn't MATCH the macro-context of the document to the query context — you
didn't cover the entity inside the query with the proper context.

**Topical Map**
A content network design based on semantics for achieving the topical authority
state by including overall macro contexts and semantics with publication frequency.
Has a Core Section and an Outer Section.
NOT connections between entities via attributes, NOT blog post name lists.

**Source Context**
The purpose of the brand identity, and how the brand monetizes its content.
It represents the main focus of the website and its connection to the central entity.
Must be reflected in site-wide N-grams, boilerplate, and main content on every page.

**Central Entity**
The entity that appears in every sub-section of the semantic content network —
in both main content/macro context and supplementary content/micro context.
Gives its main and minor attributes to the core and outer sections of the topical map.
Always appears in anchor texts with a synonym value.

**Central Search Intent**
The unification of the Source Context with the Central Entity. Processed in both
the outer and core sections of the topical map. Connected to the macro-context and
micro-context of every content brief in the semantic content network.

**Core Section of the Topical Map**
The unification of the Source Context with the Central Search Intent.
Focuses on a specific MAIN ATTRIBUTE of the central entity — the attribute that
comes from the source context. Example: if the source is an affiliate for electric
car chargers, the main attribute is "quality"; derived attributes are "durability,
charge time, maintenance."

**Outer Section of the Topical Map**
Improves overall historical data. Focuses on MINOR attributes of the central entity.
Propagates trust and quality signals to the core section via links or linkless
connections. Example: for a Visa Consultancy, the outer section covers culture,
language schools, and geography of countries — NOT visa processes.

**Main Content**
The main part of the web document. Processes the specific context-terms and topical
entries. Satisfies MAJOR query needs. Touches minor query needs only with the
contextual border. This is where the macro-context is processed. Does NOT involve
many internal links unless inside the macro-context.

**Supplementary Content**
Touches micro-contexts, sub-topics, and minor entities/attributes. Connected to
the macro-context of the page but always processed with a connection to ANOTHER
macro-context. Provides better association and "Neighborhood Content" between
different topical map segments. Contains more internal links and contextual bridges.
NOT the same as supporting content. NOT a section you add for completeness.

**Contextual Coverage**
The amount of web page document section for a specific context compared to other
sections. Heavy processing of one context dilutes the prominence of others. Light
processing reduces relevance. Connected to Passage Indexing/Ranking. Must not
break the Contextual Vector.

**Contextual Flow**
The ORDER in which contextual vectors are processed in a web document. Processing
the same things in a different order creates different possible click satisfaction
scores. Heading formats, heading words, and heading hierarchies change context
priority. Determines what is processed under what, and changes the weight of
contextual connections.

**Contextual Hierarchy**
Determines which contextual vector comes before which. Represented by typography,
visuals, and coverage of specific sub-sections. Changes the weight of internal
links for relevance and PageRank pass. Changes the macro-context of the web page.
Connected to Contextual Flow.

**Contextual Border**
The border between macro and micro context sections on a web document. Provides a
SLOW TRANSITION from main content to supplementary content. Provides a grouper
question to deepen the main context while connecting to side-topics.

**Contextual Bridge**
Hypertext or linkless connections in a semantic content network. Provided by
aligning consistent information WITHOUT a link, or WITH a hypertext for providing
contextual connection between two different topical map nodes. Focuses on query
paths, sequences, and correlative queries.

**Macro Semantics**
Semantics for the main and prominent parts of web document relevance and context —
comes from headings, anchor texts, site-wide N-grams. Evaluates: what site-wide
N-grams can appear? Most used nouns, adjectives, predicates site-wide? Most used
question formats? How heading vectors are constructed? What are the first words of
paragraphs — do they answer or use rhetoric without information?

**Micro Semantics**
Semantics for micro-adjustments for context and relevance — single word changes,
punctuation, word-order. Involves sequence modeling, word-by-word optimization of
documents for higher relevance and responsiveness. Includes discourse integration
optimization and sequence-to-sequence connections for changing contextualization.
Also involves optimization of visuals with subject and object entities.

**Relevance**
The Information Retrieval Score from text processing methodologies — term saturation,
length normalization, co-occurrence matrix construction, BM25, TF-IDF, GloVe,
Word2Vec. Shows overall connection. The "Blind Librarian" state of the search engine.

**Responsiveness**
The Information Extraction Process of search engines for giving a direct answer.
Even without relevance, there may be a quality answer. Requires query-question-answer
pairing and indexing. Connected to Passage Indexing/Ranking.

**Represented vs Representative Queries**
A search term has representative and represented query versions for expressing itself.
Search engines do NOT give the full weight of relevance to the terms put in the
search bar. The relevance of search bar terms is distributed based on the
REPRESENTATIVE queries mainly. Example: "board vision" and "vision board" give
different results because they have different contextual connections.

**Contextual Relevance**
Contextual appropriateness and significance of terms for the given context to reflect
the meaning of the term in that context. Example: the noun "love" might carry a
sexual or religious context depending on surrounding terms.

**Vastness-Depth-Momentum**
Go faster, go deeper, go wider. If one is missing, complete its missing effect by
increasing another. A bigger topical map (vastness), or deeper content briefs for
a small topical map (depth), or more frequent publication (momentum).

**Publication Frequency / Momentum**
The frequency of major content updates or new content publications. Purpose is to
be prioritized for crawling rate, indexation, and earlier rankings. Takes the
attention of the search engine content exploration systems.

**Content Configuration**
Process of optimizing relevance and responsiveness continuously according to changed
semantic distances and similarities of query terms. Comes from changed query
semantics. Requires refreshing and reconfiguring the topical maps.

Generate a semantically structured page brief for any web page type, applying

vectors, internal linking, Contextual Borders, and Contextual Bridges.

- Service Pages
- Blog Posts
- Product Pages
- Landing Pages
- Informational / Educational Pages

## Execution Steps

### Step 1 — Establish Source Context + Central Entity Alignment

Before writing a single heading:

- Define the **Source Context**: What is the brand's monetization purpose?
  How does this page connect to how the brand converts users to clients?
- Define the **Central Entity**: Which entity appears throughout this entire page
  AND throughout the site-wide content network?
- Define the **Central Search Intent**: The unification of Source Context + Central Entity
- Determine which section of the topical map this page belongs to:
  - **Core Section** → focuses on a MAIN attribute of the central entity
  - **Outer Section** → focuses on MINOR attributes; propagates trust to core section
- Identify the specific attribute of the central entity this page processes
- Check: Does the page layout and web components signal the correct conversion point
  for the search behavior? (e.g., does a quote page have a form? Does a calculator
  page have a calculator?) If not, flag it — content alone cannot rank for commercial
  queries without the corresponding functional components.

### Step 2 — Query Semantics Analysis

Determine the query's represented and representative versions:
- What is the **represented query** (exact terms users type)?
- What are the **representative queries** (how the search engine actually interprets it)?
- What is the distribution of probabilities across query variations?
- What are the MAJOR query needs (main content must satisfy these 100%)?
- What are the MINOR query needs (supplementary content touches these)?
- What are the possible and related search activities behind this query?
- Which **predicate** best signals the overall context?
  (Predicate choice changes all surrounding word sequence probabilities)

### Step 3 — EAV Depth Planning

Following Koray's battery example principle — depth is non-negotiable:

- Define the **primary entity** of this page
- List ALL relevant attributes of this entity (not just the obvious ones)
- For each attribute, list the specific VALUES (factual, verifiable data points)
- Identify DERIVED attributes from the main attributes
- Plan how attributes connect BACK to each other — disconnected attributes dilute
  context and topical consolidation
- Determine the correct attribute depth: does this page need all attributes,
  or should some attributes belong to separate pages in the topical map?
- WARNING: Do NOT include an attribute on this page if it requires a different
  macro-context. That attribute belongs to its own topical map node.

### Step 4 — Contextual Flow Design

Design the ORDER of contextual vectors before writing headings:

- What is the macro-context of this page? (The single dominant context)
- In what ORDER should contexts be processed to maximize click satisfaction score?
- Which context is heaviest and must appear earliest?
- Where does the Contextual Border sit — where does main content end and
  supplementary content begin?
- How will the page transition SLOWLY from macro-context to micro-contexts?
- What is the Contextual Bridge to the next topical map node?
- WARNING: Changing the order of sections changes the relevance of the entire document.
  Do not sequence randomly. Every ordering decision has a semantic consequence.

### Step 5 — Heading Vector Construction

Following Koray's macro semantics principles:

- H1 appears ONCE — it must contain the primary keyword and define the macro-context
- Headings are NOT for design — they signal which sections are relevant to which queries
- Heading words, formats, and hierarchy change context priority
- The first words of paragraphs matter — do they ANSWER the question or use
  rhetoric without information? Write headings that answer, not that tease.
- Consider: Could Google use this H1 as the meta title? Write it accordingly.
- Heading vectors across the page must be coherent when read as a list alone —
  the heading sequence must tell the story of the macro-context from top to bottom
- H1–H3 sufficient for most pages; only use H4–H6 for genuinely complex documents

### Step 6 — Main Content vs Supplementary Content Separation

This is one of the most critical decisions. Apply Koray's exact definitions:

**Main Content sections must:**
- Process the macro-context of the document
- Satisfy MAJOR query needs
- Cover the primary entity with its MAIN attributes and values
- Include context-terms, topical entries, and main entities
- Provide a proper connection to query contexts and semantics
- Contain minimal internal links (only if within the macro-context)

**Supplementary Content sections must:**
- Touch MICRO-contexts and minor entities/attributes
- Always maintain a connection to the page's macro-context
- Always process topics with a connection to ANOTHER macro-context (bridging)
- Contain the majority of internal links and contextual bridges
- Improve "Neighborhood Content" cohesion between topical map segments

**Contextual Border section must:**
- Sit between Main Content and Supplementary Content
- Provide a SLOW TRANSITION — not a hard stop
- Use a grouper question that deepens the main context while opening to side-topics

### Step 7 — Micro Semantics Notes for Writer

Provide sequence modeling guidance:
- Which terms need to appear in subject position vs object position?
  ("Teacher yelled students" distributes relevance differently than
  "Students are yelled by Teacher")
- Which predicates signal the correct overall context?
- Which word sequences must be preserved to maintain contextualization?
- What are the first words of key paragraphs — do they directly answer?
- Are there boolean questions to use? Antonyms for contextual bridges?
- What are the image subject entities (most concrete/prominent object)?
- What are the image object entities (context-interpreting elements)?

### Step 8 — Internal Linking Architecture

Apply Koray's internal linking principles:
- Plan the hierarchy of internal links BEFORE writing — not afterward
- Supplementary Content carries the majority of internal links
- Main Content internal links are only valid if inside the macro-context
- Each link communicates which page meets which search intent
- Anchor text must always use synonym values of the Central Entity
- Contextual Bridges (linkless associations) must be planned alongside hypertext links
- Define which pages in the topical map this page receives links FROM
- Define which pages this page links TO, with exact anchor text variations

## Page Type Guidance — Koray's Principles Applied

### Service Pages
- Source Context dominant: The page must reflect how the brand converts search users
  to clients — the web components (form, phone, booking) must be present
- Macro-context: The service itself + the specific attribute from Source Context
- Main Content: Define the service with full EAV depth → How it works (process as entity
  with attributes) → Outcomes (attributes + specific values, not vague claims)
- Contextual Border: A grouper question bridging service process to service context
  (e.g., service area, regulation, certification)
- Supplementary Content: Location entities, related services (minor attributes),
  regulatory/certification context, comparison to alternatives
- FAQ: Targets "near me", "how much", "how long", "what is included" representative queries
- Schema: LocalBusiness + Service
- WARNING: Without a quote form or contact mechanism, this page cannot rank for
  transactional queries regardless of content quality

### Blog Posts
- Represented query is usually informational — but representative queries may be commercial
- Macro-context: Directly answers the represented query. H1 mirrors the query exactly.
- Main Content: Full EAV depth on the primary entity → process from definition through
  all major attributes → connect attributes back to each other (not isolated)
- Contextual Border: Bridges from informational context toward commercial context
  (connects the topic back to the Source Context and Central Search Intent)
- Supplementary Content: Related sub-topics, internal links to commercial pages,
  Contextual Bridges to topical map nodes in the Core Section
- FAQ: Representative query extensions — what the user really needs beyond the
  represented query
- Schema: Article + FAQPage
- CRITICAL: Blog posts in the Outer Section must propagate trust signals to the Core
  Section via internal links with Central Entity synonym anchor texts

### Product Pages
- Central Entity IS the product — EAV depth is the entire page strategy
- Every attribute (size, material, compatibility, warranty, voltage, capacity) must
  have a SPECIFIC VALUE — not a vague descriptor
- Macro-context: The product + its main attribute from the Source Context
- Main Content: Product definition → ALL main attributes with specific values →
  Use cases (attributes connecting product to user context)
- Contextual Border: Bridges from product attributes to comparison context
- Supplementary Content: Comparison to alternatives, compatibility context,
  maintenance context, Contextual Bridges to related product nodes
- FAQ: "Does X work with Y", "what size do I need", "is X worth it" — these are
  representative query variants around the product's attributes
- Schema: Product + Review + Offer
- WARNING: Every claim must have a verifiable value — "high quality battery" is
  not EAV coverage. "LiFePO4 chemistry, 48V, 20Ah, 1000+ cycle life" is EAV coverage.

### Landing Pages
- Source Context = the entire page. Every element must connect to the conversion path.
- H1 must match the traffic source language exactly (ad copy, email subject)
- Contextual Flow: Problem (major query need) → Solution (Source Context + Central Entity)
  → Proof (EAV values as evidence) → Offer (Conversion goal) → CTA
- No Outer Section content here — this page is pure Core Section
- Supplementary Content: Objection-handling questions only
- FAQ: Represents the objections standing between user and conversion
- Schema: WebPage + Offer
- WARNING: Minimize navigation. Every link that takes users away from this page
  is a contextual distraction and a conversion leak.

### Informational Pages
- These often belong to the Outer Section — they build historical data and propagate
  trust signals to the Core Section
- Macro-context: The full entity definition with complete EAV depth
- Main Content: Definition → All major attributes → All attribute values → Attribute
  connections to each other (not isolated attribute mentions)
- Passage Indexing readiness is critical — each section must be self-contained enough
  to rank independently as a passage
- Contextual Border: Opens the connection from pure information toward the Source Context
- Supplementary Content: Internal links toward Core Section pages (this is how trust
  propagates from Outer Section to Core Section) — use Central Entity synonym anchors
- FAQ: Long-tail representative query variants
- Schema: FAQPage + Article
- CRITICAL: Without strong internal linking from these pages toward Core Section pages,
  the Outer Section cannot fulfill its purpose of propagating trust and historical data

## Restrictions — Non-Negotiable

Directly from Koray's course definitions:

- **Never stuff entities or attributes** — more mentions do not increase topical coverage
- **Never open a new page for every topic** without understanding how it connects to the
  central entity and which topical map section it belongs to
- **Never process an attribute in a macro-context where it doesn't belong** — this dilutes
  contextual consolidation even if the information is accurate
- **Never separate connected attributes without bridging them back** — disconnected
  attribute mentions dilute context and topical consolidation
- **H1 appears ONCE** — it defines the macro-context for the entire document
- **Never use headings for visual formatting** — headings signal relevance to queries
- **Never mix Main Content and Supplementary Content** — their separation is a
  fundamental requirement, not a stylistic choice
- **Never add internal links randomly in Main Content** — they belong in Supplementary
  Content unless they fall within the macro-context
- **Never write FAQ answers that repeat Main Content** — they must extend toward
  new representative query variants
- **Never ignore the web component requirement** — a page without the functional
  component matching its commercial query cannot rank for that query

## Self-Improvement Notes

After each brief is used by a content writer, evaluate:
- Did the page rank? For which represented or representative queries?
- Was the Contextual Border placed at the right point?
- Did the Supplementary Content correctly bridge to neighboring topical map nodes?
- Were the EAV values specific enough — or were there vague descriptors?
- Did the internal linking from this page propagate trust to the Core Section?
- Were the web components present to support the commercial query signals?
- If the page underperformed: check Historical Data signals — non-quality engagement
  from 6 months prior may be the cause, not the current content configuration
- Update page type templates when SERP characteristics change after Broad Core
  Algorithm Updates — the semantic distance between queries will have shifted
