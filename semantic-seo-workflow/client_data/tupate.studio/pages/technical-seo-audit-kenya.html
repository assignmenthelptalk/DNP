What a Technical SEO Audit Checks on a Kenyan Business Website

A comprehensive technical SEO audit examines eight categories of technical factors that either support or undermine a Kenyan website's ability to be crawled, indexed, and ranked by Google. Each category contains multiple individual checks — the following breakdown covers every element Tupate Studio's audit addresses.

1. Crawlability

robots.txt analysis: Your robots.txt file instructs Google's crawlers which pages to access and which to ignore. A correctly configured robots.txt allows Google to crawl all publicly important pages and blocks only low-value content such as admin dashboards, user account pages, and search result pages. The most damaging crawlability error Tupate Studio encounters on Kenyan websites is a robots.txt containing Disallow: / — which blocks Google from crawling the entire website. This typically occurs when a developer copies the staging site's robots.txt (which correctly blocks crawlers from the test version) to the production server without changing it. The site operates visibly, but Google has been told to ignore it entirely.

XML sitemap audit: The XML sitemap is a file listing all URLs on your website that Google should discover and index. The audit checks: is the sitemap submitted to Google Search Console? Does it include all important public pages? Does it exclude 404 error pages, redirect chains, and noindex pages? Are all sitemap URLs returning 200 HTTP status codes, not 301 redirects or 404 errors? A sitemap listing 404 URLs or pages with redirect chains wastes crawl budget and confuses Google's indexation of your site.

Crawl budget assessment: For websites with 100+ pages, Google allocates a finite crawl budget — a number of pages it will crawl in a given period. If that budget is consumed by low-value pages (paginated search results, URL parameter variations, printer-friendly page duplicates, user session ID URLs), important pages may not be crawled frequently enough to reflect recent updates. The audit identifies crawl budget waste and provides specific directives for robots.txt or canonical tag fixes.

JavaScript rendering check: Single-page application (SPA) architecture — common in React and Angular-built websites — sometimes requires JavaScript execution to render visible content. Google's crawler does not process JavaScript in real time; it queues JavaScript rendering. If critical content on your website exists only within JavaScript-rendered components, Google may not index that content or may index a version of it days behind its publication. The audit identifies whether your website has JavaScript rendering dependency issues.

2. Indexation

Google Search Console Coverage report: This report shows exactly how many of your website's pages are indexed in Google's database, why specific pages are excluded, and what errors are preventing indexation. The audit analyses every coverage report category — Submitted and indexed, Excluded (by noindex, by canonicals, by crawl errors), and Errors — and prioritises the fixes by ranking impact.

Canonical tag audit: Canonical tags tell Google which URL is the definitive version of a page when multiple URLs serve identical or near-identical content. The audit checks: are self-referencing canonicals correctly set on all pages? Are there any canonical tags pointing to the wrong URL — a common WordPress migration error where canonicals still point to the old domain? Are there conflicting canonical signals between HTTP header canonicals and HTML tag canonicals?

noindex tag audit: The <meta name="robots" content="noindex"> tag instructs Google not to include a page in its index. The most damaging manifestation of this error on Kenyan websites: a developer sets noindex on all pages during development to prevent Google from indexing the unfinished site, then forgets to remove it before launch. Google obeys the instruction and none of the site's pages appear in search results — ever.

Duplicate content detection: The same content appearing at multiple URLs dilutes ranking signal. Common sources on Kenyan websites: www.site.co.ke and site.co.ke both serving content without a permanent redirect to one canonical version; http:// and https:// both resolving without HTTPS redirect; /page/ and /page (with and without trailing slash) both serving the same content; URL parameter combinations such as /products?sort=price&order=asc generating thousands of URLs with duplicate product listings.

Thin content identification: Pages with under 200 words and no unique value contribute negatively to Google's overall quality assessment of your website domain. The audit identifies all thin content pages and recommends whether to expand them, consolidate them via canonical tags, or noindex them to remove their drag on overall site quality.

3. Core Web Vitals

LCP (Largest Contentful Paint): Time for the largest visible content element to become visible to the user. Target: under 2.5 seconds. The most common LCP failures on Kenyan websites are uncompressed hero images (800KB–2MB JPEGs served without WebP optimisation), slow hosting server response time (TTFB over 800ms — common on Kenyan shared hosting), and render-blocking JavaScript in the document head.

INP (Interaction to Next Paint): Total delay from a user's tap or click to the page's visual response. This replaced FID (First Input Delay) as a Core Web Vitals metric in March 2024 and measures across all interactions on a page, not just the first one. Target: under 200ms. Fails due to heavy JavaScript frameworks blocking the browser's main thread, analytics and tag manager scripts that run on every user interaction, and WordPress plugin JavaScript conflicts that queue multiple event listeners for the same elements.

CLS (Cumulative Layout Shift): Unexpected visual movement of page content as elements load. Target: under 0.1. Fails on Kenyan websites most commonly due to images without explicit width and height attributes (browser cannot reserve space before the image loads), web fonts that load after text has begun rendering (text shifts as the correct font replaces the fallback), YouTube or Google Maps embeds that resize on load, and cookie consent banners that appear without pre-reserved vertical space.

4. Mobile Usability

Google Search Console Mobile Usability report: Lists every page on your website that fails Google's mobile usability criteria, with the specific error type for each page. The audit reviews this report in full and correlates it with the pages' ranking performance to prioritise fixes by commercial impact.

Specific checks: Text too small to read (below 12px); clickable elements too close together (touch targets below 44px spacing); content wider than screen (causing horizontal scrolling); Flash content; intrusive interstitials blocking the main content on mobile.

5. HTTPS and Security

SSL certificate validation: Is HTTPS active and correctly configured? Are there mixed content errors where an HTTPS page loads HTTP resources (images, scripts, stylesheets) — which Chrome marks as insecure? Is the SSL certificate current (not expired)? Is it correctly covering all subdomains including www?

Security headers: HSTS (HTTP Strict Transport Security) header forces all browser requests to use HTTPS, preventing protocol downgrade attacks. Content Security Policy and X-Frame-Options headers reduce attack surface. These do not directly affect rankings but protect the website's domain authority from being compromised through security vulnerabilities.

6. Structured Data (Schema Markup)

Schema type coverage: The audit checks which schema types are currently implemented (LocalBusiness, Organization, Service, Product, Article, FAQPage, BreadcrumbList) and identifies which pages are missing schema that would benefit rankings or rich result eligibility.

Schema validation: Every schema block is validated through Google's Rich Results Test. Common errors on Kenyan websites: incomplete LocalBusiness schema missing opening hours, area served, and telephone; malformed JSON with syntax errors that invalidate the entire block; multiple SEO plugins outputting competing schema blocks for the same page — a common WordPress-specific problem that the audit specifically identifies.

7. Internal Linking

Orphan page detection: Pages with zero internal links pointing to them cannot be discovered by Google through normal website navigation — they are only reachable if submitted directly in a sitemap. Orphan pages are common on Kenyan websites after content reorganisations where old pages are left live but disconnected from the navigation.

Link equity flow analysis: Are the most commercially important pages — services pages, product category pages, high-conversion landing pages — receiving the most internal link equity from surrounding pages? Or is PageRank being diluted to low-value pages with no commercial purpose?

Anchor text audit: Internal links using generic anchor text ("click here," "read more," "learn more") pass no keyword relevance signal to the linked page. Descriptive anchor text using target keyword phrases passes both relevance and contextual information. The audit identifies all generic anchor text and prioritises replacement.

8. Page Speed

TTFB (Time to First Byte) measurement: TTFB measures how long the browser waits for the first byte of response from the server after making a request. Google's recommended TTFB is under 800ms; optimal is under 200ms. Kenyan shared hosting providers — Truehost, Sasahost, Kenya Website Experts — operate shared servers with 300–600 accounts on a single server. At peak usage times, TTFB from these servers reaches 1,500–3,000ms. This alone exceeds Google's LCP threshold of 2.5 seconds before any page content has been transferred.

Render-blocking resources: CSS and JavaScript files placed in the HTML <head> block the browser from rendering any visible content until those files are downloaded and parsed. The audit identifies all render-blocking resources and provides specific preload, defer, and async directives to eliminate the blockage.

Google PageSpeed Insights analysis: Mobile and desktop scores with each failing element identified by the Lighthouse audit tool — uncompressed images, unused CSS, unused JavaScript, long main-thread tasks, excessive DOM size. The audit translates these technical outputs into plain-language recommendations prioritised by ranking impact.

Core Web Vitals for Kenyan Websites: LCP, INP, and CLS Explained

Core Web Vitals are Google's three performance metrics that form the Page Experience ranking signal, applied since June 2021. Websites that PASS all three metrics receive a ranking advantage. Websites that FAIL are at a measurable disadvantage against competitors who pass. For Kenyan business websites competing on Google.co.ke, this is a direct, actionable ranking factor — not a theoretical future consideration.

LCP (Largest Contentful Paint) — the visibility metric:

LCP measures the time from page navigation to when the largest visible content element — typically the hero image or the H1 headline — becomes visible in the browser viewport. Google's threshold: Good is under 2.5 seconds. Needs Improvement is 2.5–4 seconds. Poor is over 4 seconds.

Kenya-specific causes of LCP failure: the most common is an uncompressed hero image — a 1.2MB JPEG downloaded over Safaricom 4G takes 0.64 seconds to download on a 15Mbps connection, before the browser begins rendering. Combined with a slow Kenyan shared hosting TTFB of 1.5 seconds, total LCP is 2.14 seconds before the image even begins painting — marginal at best, failing at worst. The fix: compress hero images to under 150KB in WebP format, serve them with the correct srcset for the visitor's screen size, and use a Cloudflare CDN to reduce server TTFB from the Nairobi edge node.

Additional LCP failure causes in Kenya: render-blocking JavaScript in the document head (the browser cannot render any content until all head JavaScript is parsed); lack of CDN (content served from a single Kenyan server has higher latency for users in Mombasa, Kisumu, Nakuru, or Eldoret than from a Cloudflare edge node); large DOM size — WordPress pages with Elementor can have 3,000–5,000+ DOM nodes, slowing all browser rendering operations.

INP (Interaction to Next Paint) — the responsiveness metric:

INP replaced FID (First Input Delay) as a Core Web Vitals metric in March 2024. Where FID measured only the delay on the first user interaction, INP measures the worst-case interaction delay across the entire page session — every button tap, every dropdown select, every form field click, every menu interaction. Google's threshold: Good is under 200ms. Needs Improvement is 200–500ms. Poor is over 500ms.

INP failure causes in Kenya: heavy JavaScript frameworks that keep the browser's main thread busy with processing tasks, preventing it from responding to user interactions. WordPress plugin JavaScript files that run large event listeners on every scroll event and every click. Google Tag Manager containers with 15+ tags firing on every page load, each running JavaScript on the main thread. Third-party chat widgets and tracking scripts competing for main thread processing time.

The fix: reduce the total JavaScript payload, defer all non-critical scripts until after the page is interactive, identify and eliminate JavaScript tasks longer than 50ms (Long Tasks) that are directly causing INP failures, and consider whether every third-party script on the page justifies its INP cost.

CLS (Cumulative Layout Shift) — the visual stability metric:

CLS measures unexpected visual shifts in page content as elements load — content moving, jumping, or being displaced by late-loading elements. Google's threshold: Good is under 0.1. Needs Improvement is 0.1–0.25. Poor is over 0.25. A CLS score of 0.25 means visible content shifted by 25% of the viewport area during page load — a jarring, disorienting experience on a Kenyan mobile screen.

CLS failure causes on Kenyan websites: images without explicit width and height HTML attributes (the browser reserves no space for the image; when it loads, it pushes all content below it down by the image height); web fonts that load 1–2 seconds after initial text rendering (text re-renders in the new font, shifting all surrounding content); YouTube embeds that render at a default size and then resize to match their container; cookie consent banners and promotional pop-ups that appear without pre-reserved space, pushing the entire page down; ads from advertising networks that load their actual dimensions asynchronously.

How to measure Core Web Vitals for your Kenyan website: Google Search Console Experience → Core Web Vitals shows field data — actual measurements from Chrome users visiting your site. PageSpeed Insights (pagespeed.web.dev) shows lab data — a controlled Lighthouse audit. Both are needed for a complete picture: field data reflects real-world Kenyan user conditions; lab data identifies specific technical elements causing failures. Tupate Studio's audit reviews both data sources for the top 10 most visited pages on your site and provides specific element-level fixes for every failure.

Crawlability and Indexation Issues That Kill Kenyan Website Rankings

Crawlability and indexation problems are the most severe technical SEO issues a Kenyan website can have — they prevent rankings entirely, for all pages affected, regardless of content quality, backlink profile, or any other ranking factor. And they are invisible without a technical audit.

Real examples from Kenyan websites Tupate Studio has audited:

robots.txt blocking the entire site: A Nairobi professional services firm launched a redesigned website. Traffic dropped to zero within 3 weeks of launch. Six months later, having invested in content and paid social media, they requested an audit. The cause: the developer had deployed the development server's robots.txt to the production server without modification. The robots.txt contained Disallow: / — Google was blocked from crawling every page. Google's index still contained stale cached versions of the old site from before the redesign. The new site had zero indexed pages and zero rankings. Fix time: 24 hours after removing the disallow directive and submitting a sitemap to Google Search Console.

noindex on all pages: A Mombasa hospitality business had a website that appeared normally to human visitors but was invisible on Google. The developer had added <meta name="robots" content="noindex, nofollow"> to the site's global header template — intended for the staging environment — and it was never removed before launch. Google correctly obeyed the instruction and indexed no pages. The business owner had been attributing zero organic traffic to "Google hasn't found us yet" for 8 months.

Non-canonical www/non-www duplicates: Both www.company.co.ke and company.co.ke were serving full content without either redirecting to the other. Google indexed both versions, splitting the ranking signal between two URLs. Neither version accumulated the full authority of inbound links and engagement signals. Fix: a 301 permanent redirect from the non-preferred version to the canonical version, confirmed in Google Search Console.

HTTP URLs not redirected after HTTPS migration: A Nairobi law firm migrated their website from HTTP to HTTPS. Their external directory listings, profile pages on the LSK directory, and inbound links from news articles all still pointed to HTTP URLs. No 301 redirects were in place. Google's PageRank associated with those HTTP inbound links was not being transferred to the HTTPS pages. Rankings were significantly lower than they should have been for a firm of their profile.

Crawl error types in Google Search Console and what each means for Kenyan websites:

404 Not Found errors: A page URL exists in the sitemap, in internal links, or has external links pointing to it — but returns a 404 when visited. Inbound link equity is not transferred through a 404. Internal PageRank flow is disrupted. The fix: either restore the page at its original URL, or implement a 301 redirect to the most relevant current page.

Redirect errors: A redirect chain — URL A redirects to URL B redirects to URL C — passes less link equity than a direct 301 and adds load time on each hop. A circular redirect — URL A redirects to URL B redirects back to URL A — produces an error and makes the page uncrawlable.

Server errors (500): The hosting server returned an error when Google tried to crawl the page. This is typically caused by a database timeout on Kenyan shared hosting under load, a PHP memory limit error on a WordPress site, or a misconfigured server configuration. Pages returning 500 errors are not indexed.

Soft 404 errors: A page returns a 200 HTTP status (indicating success) but displays "Page not found" or similar content. Google correctly identifies these as effectively 404 pages and deprioritises them, but the 200 status means they consume crawl budget without contributing to the site's indexed content.

Schema Markup for Kenyan Business Websites: Types and Implementation

Schema markup is structured data added to a website's HTML that communicates content type and meaning to Google in machine-readable format. It enables Rich Results in Google's search listings — star rating displays, FAQ expandable sections, product price and availability, event dates, recipe information — and it improves click-through rates from search results by making your listing visually richer than competitors without schema. Despite its clear ranking and CTR benefits, schema markup is one of the most under-implemented technical SEO elements on Kenyan business websites.

The schema types Tupate Studio implements for Kenyan businesses and their specific benefits:

LocalBusiness schema: The foundational schema type for any Kenyan service business. Tells Google your business name, physical address, telephone number, opening hours, geographic area served, price range, and business category — all in machine-readable format that feeds directly into Google's local search results and Google Business Profile. For Nairobi businesses, the areaServed property can list specific Nairobi districts: Westlands, Upper Hill, Kilimani, CBD, Industrial Area, Karen, Gigiri. Sub-types include MedicalBusiness for dental and medical clinics, LegalService for law firms, AccountingService for accounting firms, Restaurant for food businesses, and 40+ other specialised types.

Organization schema: Used on the homepage and About page of corporate websites. Adds the organisation's logo (used by Google's Knowledge Panel), founding date, social media profile URLs (used to confirm official social accounts in search results), and the organisation's parent company relationship for subsidiaries.

Product schema: For e-commerce and product pages. Adds product name, price, availability (InStock/OutOfStock/PreOrder), currency (KES for Kenyan stores), product image, and AggregateRating (star rating from customer reviews) directly to Google search results. A product search result showing "Ksh 4,500 — In Stock — 4.8 stars (23 reviews)" attracts significantly more clicks than a generic blue text link.

FAQPage schema: For pages with FAQ sections — including every FAQ section on every Tupate Studio page. FAQPage schema enables expandable FAQ dropdowns directly in Google's search results, dramatically increasing the visual footprint of your search listing and providing direct answers to searcher questions before they even click through. Every FAQ section on a Kenyan business website should have this schema implemented — the CTR increase is consistently significant and the implementation effort is low.

BreadcrumbList schema: Shows the page hierarchy in the URL display in Google's search results — "Tupate Studio > Services > Technical SEO Audit" instead of just the URL. Improves user orientation and click confidence for visitors evaluating search results.

Article / BlogPosting schema: For blog content and news articles. Adds publication date, last modified date, author name, and publisher information. Publication date in search results helps news-adjacent content appear in Google's Top Stories and establishes recency for time-sensitive topics.

Service schema: For individual service pages — connects each service to the LocalBusiness entity, specifying the service name, description, provider, and geographic area. Particularly important for Tupate Studio pages connecting web design and SEO services to Kenya as the service area.

Implementation method: Tupate Studio implements all schema using JSON-LD in <script type="application/ld+json"> tags — Google's recommended method, cleanest implementation, and easiest to validate and maintain. Schema is not embedded in HTML element attributes (Microdata) or RDFa, which are harder to maintain and validate.

Common schema errors on Kenyan websites: Incomplete LocalBusiness schema missing opening hours, telephone, or areaServed (reducing its utility for local search); malformed JSON with missing commas, unclosed brackets, or incorrect property types that cause the entire schema block to be ignored; conflicting schema outputs from multiple WordPress SEO plugins installed simultaneously (Yoast and RankMath both outputting LocalBusiness schema with contradictory values — Google discards the conflict).

Website Speed Audit: How Hosting in Kenya Affects Your Google Scores

Kenyan web hosting quality is one of the most significant and most overlooked technical SEO variables for Kenyan business websites. The server that hosts your website determines your TTFB — and TTFB is the first component of LCP, which is the first Core Web Vitals metric Google measures. A slow Kenyan server fails your rankings before your content, keywords, or backlinks have any opportunity to help.

TTFB (Time to First Byte) — the foundational speed metric: TTFB measures the time from a browser's HTTP request to the server to the moment the first byte of the server's response is received. Google's target: under 800ms for good LCP. Optimal: under 200ms. Every millisecond of TTFB is added directly to LCP — a 1,500ms TTFB means LCP starts at 1.5 seconds before a single image or text element has loaded.

Kenyan shared hosting reality: The most common Kenyan web hosting providers — Truehost Kenya, Sasahost, Kenya Website Experts, and their resellers — operate shared hosting environments where 300–600 customer accounts share a single server's CPU, RAM, and disk I/O. During peak hours (9am–6pm Nairobi business hours), when multiple high-traffic sites on the same server receive simultaneous visitors, TTFB for a typical shared hosting account regularly reaches 1,500–3,000ms. At 1,500ms TTFB, achieving Google's LCP target of under 2.5 seconds requires the entire remaining page rendering — including image downloads, JavaScript execution, and layout calculation — to complete in under 1 second. On a Kenyan mobile device on 4G, this is nearly impossible.

On Tupate Studio's monthly SEO retainer, mini-audits using Google Search Console and PageSpeed Insights are performed monthly, with a comprehensive full audit conducted quarterly — ensuring technical issues are caught and fixed within weeks of appearing, not discovered months later.

Can you audit a WordPress website in Kenya?

Yes — Tupate Studio audits WordPress websites and any other platform. Every technical issue we identify is platform-agnostic: crawl errors, indexation problems, Core Web Vitals failures, and schema errors exist on WordPress sites and custom sites alike. For some technical issues — Core Web Vitals failures caused by plugin overhead, schema conflicts from multiple SEO plugins, mobile usability errors from theme responsive breakpoint failures — the most effective fix recommendation will be migrating from WordPress to a custom-built codebase. We are transparent about this in the audit report when it applies.
